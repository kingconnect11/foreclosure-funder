## WHAT TO BUILD THIS WEEK (Phase 0)

### Priority 1: Supabase Project Setup

1. Create a new Supabase project (free tier is fine for now)
2. Implement the database schema from FOUNDING_ARCHITECTURE.md Section 8. Core tables needed immediately:
   - `profiles` (extends auth.users — role, subscription_tier, deal_room_id)
   - `deal_rooms` (agent/group containers)
   - `markets` (geographic regions with scraper config)
   - `properties` (the core property data)
   - `court_research` (per-property, can be empty for now)
   - `investor_pipeline` (per-investor per-property CRM stages)
   - `investor_preferences` (from onboarding — can be empty for now)
   - `recommendation_scores` (computed, can be empty for now)
3. Set up enum types: user roles (`super_admin`, `admin`, `investor`), subscription tiers (`free`, `standard`, `premium`), pipeline stages (see Section 11 for the 12 stages)

### Priority 2: Supabase Auth

1. Enable email/password auth
2. Create a trigger: when a new user signs up in `auth.users`, auto-create a row in `profiles` with role `investor` and tier `free`
3. Philip's account: super_admin role
4. Mike's account: admin role, linked to a deal room
5. Sign in with Apple and Google can wait until the mobile app phase

### Priority 3: Row-Level Security (RLS)

This is critical. Every table with user data needs RLS policies. Key rules:
- Investors can only read/write their own `investor_pipeline`, `investor_preferences`, and `recommendation_scores` rows
- Admins can read all data where `deal_room_id` matches their own deal room
- Super admins can read/write everything
- `properties` and `court_research` are readable by all authenticated users (feature gating happens at API level based on tier, not at DB level)
- `profiles` — users can read their own profile; admins can read profiles in their deal room

### Priority 6: New Next.js App Skeleton
